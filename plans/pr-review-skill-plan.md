# PR Review Skill

## Context

ShopStack currently has two review surfaces:
- `code-structure-reviewer` agent (`.claude/agents/code-structure-reviewer.md`) — runs on the staged diff at commit-time, advisory, narrow (naming/lint/docstyle/imports/PEP8), output is a single one-paragraph warning.
- CI coverage gate (`.github/workflows/ci.yml`) — runs tests with a coverage floor.

There is no skill that reviews an already-opened PR holistically (code quality + structure + performance) and posts categorized feedback back to GitHub. Reviewers currently either run ad-hoc local review or comment by hand. The new skill closes that gap: a single `/pr-review [<PR#>]` invocation diffs the PR, evaluates it across the three axes, and writes findings back as PR comments tagged Critical / Warning / Suggestion.

## Scope

- New project-level skill at `.claude/skills/pr-review/SKILL.md` (checked into repo, shared with the team).
- Invokable as `/pr-review` (uses current branch's PR) or `/pr-review 42` (explicit PR number).
- Posts **inline review comments** on changed lines where findings are line-anchored, plus a **summary comment** for cross-cutting / architecture-level notes. Both go up as one `gh pr review` submission with `event=COMMENT` (advisory only — never requests changes or should never approve).
- Three severity buckets: **Critical**, **Warning**, **Suggestion**. Each comment body is prefixed (e.g. `[Critical] N+1 query in OrderSerializer.to_representation`).

## Review dimensions

The skill's prompt instructs the model to evaluate each changed hunk along three axes — and assign exactly one severity per finding:

1. **Code quality** — bugs, error handling gaps, dead code, unsafe assumptions, missing edge cases, security risks (injection, auth bypass, secret leakage) and multi tenant handling.
2. **Structure** — separation of concerns, cohesion, naming, public-API docstrings, import order, PEP8. Defers to `code-structure-reviewer` rules so the two surfaces stay aligned.
3. **Performance** — N+1 queries (Django `.select_related`/`.prefetch_related` misses), unnecessary loops over querysets, missing indexes for new filters, large in-memory loads, blocking calls in the request path, incorrect filters in db queries.

Severity rubric (embedded in SKILL.md):
- **Critical** — correctness bug, security hole, data loss risk, broken multi-tenant isolation, definite N+1 in a request path.
- **Warning** — likely defect, structural smell that will compound, missing tests for non-trivial branch, performance concern needing measurement.
- **Suggestion** — readability, naming, minor refactor, nit-level style.

## Critical files

- **New:** `.claude/skills/pr-review/SKILL.md` — the only file to add. Self-contained: YAML frontmatter (`name`, `description`), then the workflow below.

## Workflow encoded in SKILL.md

1. **Resolve PR number.** If `$1` provided, use it. Else `gh pr view --json number -q .number` on current branch. Fail clearly if neither resolves.
2. **Collect PR context** via `gh`:
   - `gh pr view <N> --json title,body,baseRefName,headRefName,headRefOid,files`
   - `gh pr diff <N>` for the unified diff.
3. **Analyze.** Model walks the diff hunk-by-hunk, producing a structured finding list:
   ```
   [{severity, path, line, dimension, body}, ...]
   ```
   Plus a separate `summary` array for cross-cutting notes (no path/line).
4. **Post results** in one `gh api` call to `/repos/{owner}/{repo}/pulls/{N}/reviews` with:
   - `event: "COMMENT"` (advisory only)
   - `body:` overall summary in Markdown — severity counts + cross-cutting notes
   - `comments: [...]` array with `path`, `line`, `side: "RIGHT"`, `body` for each line-anchored finding
   - Use `gh api --method POST ... --input -` with a heredoc-built JSON payload so multi-line bodies survive shell quoting.
5. **Echo a one-screen recap** to the terminal: counts by severity and link to the posted review.

## Reused tooling

- `gh` CLI (already used in the project per CLAUDE.md PR workflow).
- Severity language and import/PEP8/docstyle rules mirror `.claude/agents/code-structure-reviewer.md` so the two surfaces don't contradict.

## Out of scope (deliberately)

- Does not block merges (no `REQUEST_CHANGES`).
- Does not run tests or coverage — CI already does.
- Does not auto-fix; only comments.
- Does not review across multiple PRs in one invocation.

## Verification

1. From the repo root, run `/pr-review <PR#>` against an open PR.
2. Confirm on GitHub:
   - A single review appears with `COMMENT` status.
   - Inline comments are anchored to the correct changed lines.
   - Each comment body starts with `[Critical] / [Warning] / [Suggestion]` and names the dimension.
   - The summary body lists severity counts and any cross-cutting notes.
   - The summary body should also contain a text `Commented by Claude-Code` to indicate it was generated by the skill.
3. Run `/pr-review` with no argument on a branch that has an open PR; verify auto-resolution.
4. Run on a branch with no open PR; verify the skill exits with a clear error and posts nothing.
